1. Introduction

MBASE library's source code is written with some conventions in mind. The conventions defined for the development of the library do not cover a huge amount of content, but it is generally beneficial for writing a structural, beautiful, and semantically consistent code. If something is unspecified in the convention, then the developer is free to do it his/her own way.

2. Types of Conventions

There are four categories of conventions that are listed below:

  • 3. Common Data Types and Macros
  • 4. Structural Conventions
  • 5. Semantic Conventions
  • 6. File Structure and Comments

3. Common Data Types and Macros

The common data types to use throughout the development, will reside inside the "common.h" file. Along with data types, it also contains the necessary macros for structuring the code. Those data types must be used throughout the development.

3.1 Data Types Table

Standard(0): When using data types, the custom type aliases must be used.

Rationale: Type of the data indicates where it will be used and how it will be used. Writing a GENERIC instead of void makes more sense as what the type does. Also by writing I32 instead of int32_tsaves us a tiny amount of keyboard work.

3.2 Macros

There are some macros that are essential for library to work and semantic integrity. I am aware that the use of macros is discouraged in C++ programming. However, they are useful at times if they managed carefully.

Standard(0): Use of macro functions should be avoided unless it is compeletly necessary.

Rationale: Function macros are very difficult to debug and they pollute the local scope's namespace. The general use of macros should be a name,string substitution for repetitive statements or expressions.The general use of macros in MBASE looks like this: 

  1. #define MBASE_STD_EXPLICIT explicit
  2. #define MBASE_EXPLICIT explicit
  4. #define MBASE_STD_NAME "MBASESTD"
  5. #define MBASE_STD_VERSION "1.0.0"
  6. #define MBASE_STD_VERSION_N 100
  8. #define MBASE_STD_BEGIN namespace mbase{
  9. #define MBASE_STD_END }

For this reason, they should generally be expanded as one-liners and descriptive replacements for basic expressions.

Standard(0): All macros must start with prefix MBASE_.

Rationale: So that we won't pollute the global macro space and maintain consistency.

4. Structural Conventions

Structural conventions are conventions that define the structure of the program through following categories:

  • Source code Ordering
  • Class Content Ordering
  • Coding style.

4.1 Source code ordering

4.1.1 Code Setup

Standard(0): Every header file must have the traditional include guards not the "pragma once"

Rationale: pragma once is not a standard

Standard(0): Every header file must include "common.h"

Rationale: It contains the common types and macros.

Standard(0): The code must reside inside the namespace "mbase". This must be achieved through placing macros MBASE_STD_BEGIN and MBASE_STD_END.

Rationale: To not pollute the namespace of the user who use the library.

Example

Here is how an empty header file looks like: 

  1. #ifndef MBASE_SAMPLE_HEADER_H
  2. #define MBASE_SAMPLE_HEADER_H
  4. #include <mbase/common.h>
  6. MBASE_STD_BEGIN
  8. // code here...
  10. MBASE_STD_END
  12. #endif // MBASE_SAMPLE_HEADER_H

4.1.2 Implementation Order

In the source file, the implementation order goes as follows:

  • Include directives.
  • Macros.
  • Template meta-programming structs.
  • Class declarations.
  • Struct declarations
  • Function declarations.
  • Global variables.
  • Class definitions.
  • Struct definitions
  • Function definitions.

Since the enumeration type is missing here, it can be placed anywhere on the code. As shown in the list above, the standard goes as follows:

Standard(0): Implementation order must be preserved in every source file.

Rationale: Integrity.

4.1.3 Class Content Ordering

Disclaimer: The specification that have been made on this section, also applies to structs. Further description of behaviors is explained on page More on Class Structuring.

Firstly, Classes are basically state storage mechanisms. Its state can be copied, moved, i.e. modified etc. Classes name or the operations it does, tell something about its behavior. For example, if the class does not contain any data and just merely provides some virtual or normal methods, its behaviors can be: Stateless, Abstract Interfaceable, Callable etc. Thus, all classes have some behaviors.

Secondly, since classes operate on a state that is holding (or operates on a data that is holding), it exposes an interface (some can all API), to either manipulate, observe, create etc. data. And it achieves those behaviors via its methods. So, for this reason, we will define the behaviors of a class, and method and data ordering.

Class behaviors

The list of behaviors a class can have is as follows:

  • Default Construbtible
  • Move Constructible
  • Copy Constructible
  • Destructible
  • Equality Comparable
  • Swappable
  • Allocate Aware
  • Iterable
  • Callable
  • Copy Assignable
  • Move Assignable
  • Arithmetic Operable
  • Equality Comparable
  • Sign Comparable
  • Templated
  • Serializable
  • Abstract Interfaceable
  • Full Abstract
  • Stateless
  • Type Aware
  • Debuggable

Method Ordering

Here is the order of method groups:

  • Builder Methods
  • Operator Builder Methods
  • Iterator Methods
  • Observation Methods
  • State Modifier Methods
  • Operator State Modifier Methods
  • Non-modifier Methods
  • Operator Non-modifier Methods
  • Non-member Functions
  • Operator Non-member Functions

So, from now on, we can derive standards such as:

Standard(0): A class must have at least one behavior which is defined in behavior table.

Rationale: At least one behavior fit for every scenario.

Standard(0): Class methods must be grouped in the correct order as the order defined in method groups table.

Rationale: Readibility.

Standard(0): Unless the method is; templated, non-member(static or friend), operator non-member, the definition must be outside the class.

Rationale: Readiblity.
